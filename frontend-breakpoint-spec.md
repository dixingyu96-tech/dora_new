# 前台响应式断点规范

> 适用范围：Dora 前台工作台、会话页、专家团、资料库、定时任务等主界面
> 文档用途：可直接交付研发实现
> 版本：v1.0
> 更新时间：2026-07-16

---

## 1. 目标

本规范用于统一前台页面的响应式断点策略，避免出现：

1. 页面之间断点值各写各的
2. 组件为修局部问题不断新增临时断点
3. 同一宽度下，不同页面导航结构不一致
4. 桌面态与平板态之间只有“缩小”，没有明确布局切换

本次规范采纳的原则：

1. 断点优先按**布局模式**定义，而不是按设备名定义
2. 在布局模式之外，再补一层**桌面密度断点**
3. 尽量减少不规则断点，统一为少量高复用档位

---

## 2. 最终断点体系

### 2.1 标准断点

| 断点区间 | 模式名称 | 页面结构 |
| --- | --- | --- |
| `>= 1280px` | 宽桌面 | `tab + 侧边栏 + 内容` |
| `840px - 1279px` | 窄桌面 / 横屏平板 | `tab + 侧边栏 + 内容` |
| `600px - 839px` | 中屏模式 | `tab + 内容` |
| `< 600px` | 手机单列 | `单列` |

### 2.2 断点含义

这套断点分两层理解：

1. `840px` 和 `600px` 是**布局模式断点**
2. `1280px` 是**桌面密度断点**

换句话说：

- `840px` 决定侧边栏是否还能保留
- `600px` 决定是否彻底切到单列手机模式
- `1280px` 决定虽然都是“完整桌面结构”，但页面留白、卡片密度、最大宽度是否进入更舒适的桌面排布

---

## 3. 为什么采用这套断点

### 3.1 为什么保留 `840px`

`840px` 是前台从“完整信息架构”切到“压缩信息架构”的关键点。

在此宽度以上，通常还能稳定容纳：

1. 左侧 tab 导航
2. 右侧侧边栏或辅助区
3. 主内容区

在此宽度以下继续保留侧边栏，会明显压缩：

1. 主输入区宽度
2. 卡片宽度
3. 顶部筛选工具栏
4. 推荐标签与文件卡片可用空间

因此，`840px` 适合作为“是否保留侧边栏”的结构断点。

### 3.2 为什么保留 `600px`

`600px` 是从“中屏双区”切到“手机单列”的断点。

在 `600px - 839px` 区间，仍可保留：

1. 左侧 tab rail
2. 主内容区

但如果低于 `600px` 还继续维持双区，会造成：

1. 输入区过窄
2. 文件卡片过密
3. 卡片列表换行混乱
4. 推荐区和工具栏可用空间不足

因此，`600px` 适合作为“是否切手机单列”的结构断点。

### 3.3 为什么新增 `1280px`

如果只有 `840px / 600px` 三档，虽然布局结构够用，但桌面体验会太粗。

`840px` 和 `1440px` 虽然都属于完整模式，但视觉密度完全不同，主要体现在：

1. 页面左右留白
2. 卡片列数和列宽
3. 内容区最大宽度
4. 筛选区与工具栏的舒适度
5. 推荐模块和 hero 区的节奏

因此需要加一个 `1280px`，把完整模式再拆成：

1. 宽桌面：更舒适、更有呼吸感
2. 窄桌面：结构不变，但要收紧密度

---

## 4. 各断点下的布局规则

## 4.1 `>= 1280px`：宽桌面

### 页面结构

- 左侧：tab 导航保留
- 右侧：侧边栏 / 辅助区保留
- 中间：主内容完整展开

### 适配目标

优先保证“舒适桌面密度”，不是单纯把内容拉满。

### 研发规则

1. 页面内容区应设置合理 `max-width`，避免超宽拉散
2. 卡片区优先使用更稳定的多列布局
3. 工具栏、搜索、筛选器应尽量保持单行
4. hero / sender / 推荐区允许保留更宽留白
5. 右侧辅助区可维持完整展示

---

## 4.2 `840px - 1279px`：窄桌面 / 横屏平板

### 页面结构

- 左侧：tab 导航保留
- 右侧：侧边栏 / 辅助区保留
- 中间：主内容保留

### 适配目标

在不丢结构的前提下收紧密度。

### 研发规则

1. 内容区内边距可比宽桌面略小
2. 卡片最小宽度可以收紧，但不改结构模式
3. 顶部工具栏允许局部压缩，但优先不换行
4. 次级信息可适度减少视觉占比
5. 侧边栏宽度可保持不变，主内容区负责吸收压缩

---

## 4.3 `600px - 839px`：中屏模式

### 页面结构

- 左侧：tab 导航保留
- 右侧：侧边栏隐藏
- 主区：内容独占

### 适配目标

保住一级导航，优先让主内容可用。

### 研发规则

1. 关闭右侧侧边栏、快捷区、辅助信息区
2. 主内容区独占可用宽度
3. 页面内双栏结构优先改为单主列
4. 筛选区和工具栏允许换行或上下堆叠
5. 复杂卡片列表优先降列数
6. 输入区、文件卡片区、技能推荐区优先保留主功能，不保留多余装饰

---

## 4.4 `< 600px`：手机单列

### 页面结构

- 顶部：轻量导航
- 主区：单列内容
- 底部：底部导航优先

### 适配目标

把所有复杂工作台结构收束成单列流程，优先保证可点、可读、可输入。

### 研发规则

1. 页面全部采用单列内容流
2. tab rail 隐藏，转为顶部或底部导航
3. 筛选、搜索、操作按钮允许纵向堆叠
4. 卡片区统一降为 1 列
5. 推荐区、文件区、输入区都按手机交互重排
6. 避免横向滚动成为主交互路径

---

## 5. 组件级响应式规则

以下规则用于把页面级断点落到具体组件。

## 5.1 导航结构

### `>= 840px`

保留：

1. 左侧 tab rail
2. 右侧侧边栏 / 快捷区 / 辅助区

### `600px - 839px`

保留：

1. 左侧 tab rail

隐藏：

1. 右侧侧边栏
2. 次要辅助区

### `< 600px`

隐藏：

1. 左侧 tab rail

替代：

1. 顶部轻导航
2. 底部导航

---

## 5.2 顶部工具栏 / 筛选区

### `>= 1280px`

1. 尽量保持单行
2. 搜索、筛选、创建按钮共存
3. 输入宽度按舒适桌面处理

### `840px - 1279px`

1. 尽量保持单行
2. 可通过最大宽度、最小宽度和内容裁剪收紧
3. 不建议过早拆成两行

### `600px - 839px`

1. 允许拆成上下两层
2. 搜索与筛选区可自适应换行
3. 固定宽按钮应优先保留主按钮

### `< 600px`

1. 默认纵向堆叠
2. 每个控件宽度尽量占满
3. 操作按钮单独成行更稳

---

## 5.3 卡片列表

### 宽桌面

1. 使用稳定多列
2. 维持统一列宽和舒适间距
3. hover 投影和展开内容需预留安全空间

### 窄桌面 / 横屏平板

1. 保持多列，但减少每列最小宽度
2. 卡片信息不应因压缩产生明显跳动

### 中屏

1. 优先降列数
2. 保证单卡内部信息优先级清晰

### 手机

1. 单列
2. 卡片内容尽量顺序展开，不做复杂悬浮依赖

---

## 5.4 输入区 / Sender

### `>= 840px`

1. 输入区可保留完整附件、技能、引用入口
2. 文件卡片和技能推荐区可以横向展开

### `600px - 839px`

1. 输入区仍保留完整功能
2. 附件、技能、推荐模块需要压缩排布
3. 优先保住输入行为，不保过大留白

### `< 600px`

1. 单列输入流程
2. 附件入口、技能入口、发送按钮都要优先保证可触达
3. 文件卡片、技能推荐卡需要明显缩减密度

---

## 5.5 文件卡片 / 推荐标签 / 横向滚动元素

### 原则

1. 断点变化时优先减少列数或可见数量
2. 其次压缩宽度
3. 最后才考虑横向滚动

### 原因

这些元素在工作台里不是主内容阅读区，过度依赖横向滚动会伤害效率。

---

## 6. 推荐的媒体查询写法

## 6.1 断点常量

建议在样式系统中统一定义：

```css
--bp-desktop-wide: 1280px;
--bp-layout-full: 840px;
--bp-layout-single: 600px;
```

如果使用 JS 常量，也应同步定义一份：

```ts
export const BREAKPOINTS = {
  desktopWide: 1280,
  layoutFull: 840,
  layoutSingle: 600,
}
```

## 6.2 媒体查询建议

```css
@media (min-width: 1280px) {
  /* 宽桌面密度 */
}

@media (min-width: 840px) and (max-width: 1279px) {
  /* 窄桌面 / 横屏平板 */
}

@media (min-width: 600px) and (max-width: 839px) {
  /* tab + 内容 */
}

@media (max-width: 599px) {
  /* 单列手机 */
}
```

说明：

1. 统一使用 `840 / 600 / 1280`
2. 不建议写成 `>839`、`<600` 这种口语式条件
3. 边界值要明确，避免 839/840、599/600 出现实现歧义

---

## 7. 研发实现原则

1. 先做页面结构断点，再做组件细节断点。
2. 不允许为了修某个页面局部问题随意新增 `930px`、`1180px` 这类临时断点，除非评审确认确有必要。
3. 同一类布局在不同页面应复用同一套断点，不要一个页面 `900`，另一个页面 `1024`。
4. 宽桌面和窄桌面可以同结构，但必须允许不同密度。
5. 手机模式不要只靠缩小字体和压缩宽度来“假装适配”，要真正切成单列流程。

---

## 8. 验收清单

研发完成后，至少需要按以下宽度验收：

| 验收宽度 | 预期模式 |
| --- | --- |
| `1440px` | 宽桌面 |
| `1280px` | 宽桌面边界 |
| `1024px` | 窄桌面 / 横屏平板 |
| `840px` | 完整布局边界 |
| `768px` | 中屏 |
| `600px` | 中屏边界 |
| `599px` | 手机单列 |
| `390px` | 常规手机 |

验收重点：

1. 导航结构是否切换正确
2. 侧边栏是否按预期出现/隐藏
3. 顶部工具栏是否过挤
4. 卡片是否出现不合理截断
5. 输入区是否可正常使用
6. 文件卡片、推荐区、技能标签是否失控换行或溢出

---

## 9. 与现有代码的关系

当前项目里已经存在一些历史断点，例如：

1. `1280px`
2. `1024px`
3. `900px`
4. `640px`

后续建议：

1. 保留 `1280px`，将其正式收编为宽桌面密度断点
2. 用 `840px` 替代目前不够统一的完整布局阈值
3. 逐步减少 `900px` 这类语义较弱的局部断点
4. 将 `640px` 类手机断点逐步收敛到 `600px` 体系；如个别组件确实需要更早切换，可作为组件级补充断点，但不能替代全局断点

---

## 10. 最终结论

前台断点体系统一为：

1. `>= 1280px`：宽桌面，`tab + 侧边栏 + 内容`
2. `840px - 1279px`：窄桌面 / 横屏平板，`tab + 侧边栏 + 内容`
3. `600px - 839px`：中屏，`tab + 内容`
4. `< 600px`：手机单列

这套方案兼顾了：

1. 导航结构稳定性
2. 工作台类页面的信息密度
3. 平板与手机之间的过渡清晰度
4. 研发实现的一致性与可维护性
